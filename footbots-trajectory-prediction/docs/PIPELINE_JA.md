# パイプライン詳細

## 1. 検出

`src.train_yolo`はYOLO形式データセットの構造とラベル値を検証し、`data.yaml`を生成してUltralytics YOLOを学習します。

クラス定義は次の通りです。

| ID | クラス |
|---:|---|
| 0 | player |
| 1 | ball |

## 2. 追跡と座標変換

`src.track_video`はYOLOの`track` APIを利用し、ByteTrackまたはBoT-SORTで選手IDを関連付けます。選手位置はバウンディングボックス下辺中央を代表点とし、ホモグラフィで競技場座標へ変換します。

```csv
frame,agent_id,agent_type,x,y,score,source_track_id,det_class
0,17,1,40.1,28.5,0.91,17,0
0,0,0,52.5,34.0,0.74,3,1
```

## 3. 前処理

`src.prepare_tracks`は以下を実行します。

1. 競技場外の座標を除外
2. 観測長が短い軌跡を除外
3. 同一ID・同一フレームの重複を整理
4. 短い内部欠損を線形補間

## 4. ボール補完

`src.complete_ball_track`は次の優先順位でボール座標を補完します。

1. 実検出
2. 2つの実検出間の短区間補間
3. 推定保持者の進行方向前方に置く代理点
4. 直前速度を減衰させる外挿

各行には`ball_source`と`ball_confidence`が追加され、補完方法を追跡できます。

## 5. 全エージェント軌道モデル

`TrajectoryTransformer`は入力`[batch, time, agents, features]`を受け取ります。各ブロックは次の順に処理します。

1. エージェントごとの時間自己注意
2. 時刻ごとの社会自己注意
3. 残差接続、LayerNorm、FFN

未来位置は最後の観測位置に予測変位を累積して計算します。

## 6. ボール軌道モデル

`BallTrajectorySetBiLSTM`は、各時刻の選手集合をSet Transformerで符号化し、選手文脈とボール履歴を別々の双方向LSTMへ入力します。融合文脈から将来ボール変位を復号します。

## 7. 評価上の注意

ランダムな窓単位分割では、同一試合の隣接フレームが訓練・検証へ跨る可能性があります。正式な評価では、試合、前後半、または攻撃系列単位で分割してください。

推奨比較:

- 停止モデル
- 等速度モデル
- 単体LSTM
- 社会相互作用モデル
- 時空間Transformer
- 時間注意・社会注意を除いたアブレーション
